# Agent.md

## 目的

本仓库以后统一用 `my-project-capability-architecture.xlsx` 追踪两件事：

1. 项目当前已经具备哪些能力。
2. 每个能力现在处于 `已实现`、`部分实现` 还是 `未实现`。

任何 Agent 在做功能规划、功能开发、补齐能力、复盘完成度时，都必须优先更新这份表，不能只在对话里口头说明。

## 台账文件

- 工作簿：`/Users/jiachenwang/Desktop/ai工作室/my-project-capability-architecture.xlsx`
- 主表：`能力架构`
- 汇总表：`状态统计`

当前主表列定义如下：

- `能力域`：能力所属模块，例如 `设计工作台`、`实时协同`、`AI协同设计`
- `子能力`：最小可跟踪的能力项，要求是用户能感知、能验证的功能点
- `状态`：只能填写 `已实现`、`部分实现`、`未实现`
- `你现在的实际能力`：只写当前真实具备的能力，不写愿景，不写计划
- `代码依据`：必须填写代码、页面、接口或测试依据，尽量带文件名和行号

`状态统计` 用来汇总主表数据。更新完 `能力架构` 后，要同步确认统计没有失真。

## 状态口径

- `已实现`：代码库里已经有完整闭环，用户可以实际使用，并且能给出明确代码依据。
- `部分实现`：已经有入口、半成品链路或局部能力，但还不算完整产品能力。
- `未实现`：当前代码库中没有看到可以确认的产品化实现。

不要因为“有设计稿”“有文档”“未来要做”就标成 `部分实现` 或 `已实现`。判断依据只看当前代码库里的真实能力。

## 必须更新表的时机

- 新增功能前：先查看主表里有没有对应能力项。
- 新增功能时：如果这是新的能力点，要在 `能力架构` 新增一行。
- 完成功能后：更新该行的 `状态`、`你现在的实际能力`、`代码依据`。
- 补齐半成品功能后：把 `部分实现` 改成更准确的状态。
- 修复重大缺陷后：如果能力成熟度发生变化，也要更新这张表。
- 只做重构时：如果用户可感知能力没有变化，可以不新增能力项；但如果代码依据明显变化，可以更新 `代码依据`。

## 行级更新规则

### 1. 新增能力项

满足下面任一情况，就新增一行：

- 这是一个主表中还不存在的新功能。
- 这是一个用户可以单独感知和验证的新能力。
- 这个功能不适合硬塞进已有 `子能力`，否则会让描述变得含糊。

新增时遵循这些规则：

- 优先挂到已有 `能力域` 下；只有确实没有合适归属时，才新建 `能力域`。
- `子能力` 要写成具体能力，不要写成模糊项目名。
- 初次登记时，如果只是规划、还没做，写 `未实现`；如果已经做了一半，写 `部分实现`。

### 2. 更新已有能力项

如果本次工作只是把已有能力做完整、做稳定、做可用，就更新原有行，不重复新增。

更新时：

- `你现在的实际能力` 只保留当前事实，不保留历史描述。
- `代码依据` 写这次能力成立最关键的文件、页面、接口、测试。
- 如果一个能力跨多个文件，可以用分号分隔。

### 3. 规划下一步工作

以后做规划时，默认按这张表来排优先级：

- 先看 `部分实现`，优先把已经有基础的能力补成完整闭环。
- 再看 `未实现`，选择最影响项目价值的能力补齐。
- 对已经 `已实现` 的能力，重点检查是否缺测试、缺稳定性、缺体验补全。

## 每次任务的标准流程

每次 Agent 开始做功能类任务时，按下面顺序执行：

1. 先读取 `my-project-capability-architecture.xlsx`，找到受影响的能力项。
2. 判断这次是“新增能力”还是“完善已有能力”。
3. 完成代码或规划工作。
4. 回写 `能力架构`。
5. 检查 `状态统计` 是否仍然准确。
6. 在最终汇报中明确说明这次更新了哪些能力项。

## 填写要求

- 不写空泛描述，例如“支持更多能力”“体验更完整”。
- 不写未来时，例如“将支持”“计划支持”。
- 优先写可验证事实，例如页面入口、接口路径、测试文件、核心脚本。
- 如果没有代码依据，不要把状态标成 `已实现`。
- 一项能力只保留一个当前状态，不要在单元格里混写多个阶段。

## 失败兜底

如果当前环境里暂时不能可靠写回 `.xlsx`，也不能跳过这一步。至少要做到：

1. 明确说明这次本应更新哪一行或新增哪一行。
2. 给出完整的 5 列内容。
3. 说明为什么这次没能实际写回工作簿。

默认目标仍然是直接更新工作簿，而不是长期依赖兜底方案。

- 涉及页面入口、静态资源路径、`manifest.webmanifest`、`sw.js`、路由兜底等改动时，要默认保持 GitHub Pages 兼容。

## 本仓库的执行约束

- 这份 Excel 是能力台账的单一事实来源。
- 功能完成与否，以表里的状态和代码依据为准，不以口头描述为准。
- 以后新增功能、完成功能、补齐功能时，都要把这张表一起更新。
