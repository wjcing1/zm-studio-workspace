## 0. 一句话定义

**主版本（作品集开篇 / 简历项目栏）**：
> 建筑师的 Cursor。视觉聚类是建筑师的语法，AI 跟着读。

**电梯版（30 秒 pitch / 给非专业朋友讲）**：
> 建筑师的 Cursor。建筑师出判断，AI 接细节。

**为什么不是其他写法**：
- 不写"AI 原生协作画布"——所有人都这么说
- 不写"AI 负责实现"——过度承诺，AI 实际做的是辅助
- "Cursor"做类比锚点，让面试官 0.5 秒 get 到形态（AI 原生 IDE → AI 原生画布）
- 主版本暴露独有洞察（视觉聚类作为建筑师的语法），不可被替换到其他领域

**和 Cursor 的差异点（被追问时的 30 秒答案）**：
Cursor 处理**已经被语言化的工件**（代码）；我的产品处理**尚未被语言化的工件**（建筑师脑中靠图和位置表达的方案）。Cursor 是文本编辑器形态；我是画布形态。

## 思考演化轨迹（专属节，AI 产品岗加分项）

> 此节专门保留产品判断的反复过程，不假装"从一开始就想清楚了"。

### 演化轨迹 1：AI 是否需要消费画布坐标

- **初判**：不需要。LLM 对二维几何空间语义理解弱（区分不了 L 型 / 平行 / 包络这种描述），传坐标只是徒增 token 消耗，回答质量不会提升。结论：AI 只读节点之间的**显式连接关系**。
- **修正**：需要——但不是"几何描述"层面的坐标，而是**视觉聚类语义**。建筑师组织画布时有两层未显式表达的语法：
  1. **聚类**：相关的内容会被放近（四张图聚成一组、五段文字聚成一组）——没有连线，但空间临近本身就是分组语义
  2. **尺度排序**：从左到右、从上到下经常对应"大尺度 → 中尺度 → 小尺度"或时间顺序——空间顺序即逻辑顺序
- **当前实现**：`collectContextText` 只读节点内容文本与连接关系，**未读聚类与排序信息**（gap 已识别，见决策 D1）
- **下一步路线**：方案 C — 几何聚类（DBSCAN / 距离阈值）+ LLM 语义化贴标 + 用户可手动框选分组覆盖

**这一段的价值**：体现"在用产品的过程中发现用户行为里的隐性结构"——这是 AI PM 的核心能力，也是和普通 PM 的区分点（知道用什么算法补位）。

### 演化轨迹 2：AI 入口要不要永久可见

- **初判**：sidebar 是可有可无的 UI 元素，按需呼出即可（参考 Notion AI、Miro AI 的隐藏式入口）。
- **修正**：sidebar 不仅必须存在，而且**默认必须开启**——这是产品定位的根。
- **触发这次修正的判断**：
  > "如果 sidebar 一开始是隐藏起来的话，那么这个就说不上是 AI 原生的产品，属于是 AI 附加。"
- **底层判断标准**：
  - **AI 原生** = 用户第一眼就**看到** AI 的存在，AI 不需要被"激活"才出现
  - **AI 附加** = AI 是个隐藏功能，需要用户主动想起来才用
  - 隐藏 = 附加，常驻 = 原生
- **结论**：永久 sidebar 默认开启 + Space 选中节点的 fast path 双触发（决策 D5）

**这一段的价值**：把"AI 原生"这个被用滥的词落地成可验证的设计原则。AI 产品岗最稀缺的能力——把模糊概念变成可操作的设计判断。

## 1. 问题为什么真实存在

### 1.1 一次真实的崩溃（2026 年初）

**场景**：米兰理工建筑学院课程作业，做场地方案。

**画布状态**：堆了城市分析图、场地黑白平面图、若干参考图。

**操作**：在 Miro 内建 AI 输入"把这张黑白场地平面图渲染成彩色平面图"。

**AI 输出**：渲染图里**原场地上的建筑和树木位置全错位、道路也接不上**——本质上 AI 没在原图基础上"上色"，而是**重新画了一张和原图无关的彩色图**。

**收尾**：放弃 Miro 内建 AI，开 ChatGPT 用**自己积累的提示词模板**重新走一遍流程才得到能用的结果。

**这次失败暴露的根问题**：不是 Miro 的 AI"水平差"，而是**它没办法把画布作为有结构的上下文消费**——它只是在一个角落里跑了一个独立的图像生成 API，跟画布上的其他东西（参考图、文字说明、之前的草图）完全脱钩。

> 待补细节（不阻塞主线）：那次崩溃造成的时间损耗估算 + 同型号失败的频率。

## 3. 核心决策 × 5

> 每个决策严格遵循模板：**决策 / 否掉的备选 A / 否掉的备选 B / 当时不确定 / 事后看**。
> 演化中的决策另在"思考演化轨迹"节有独立轨迹记录。

### D1 — AI 消费画布的语义结构

**决策**：AI 消费画布的内容 = **显式连接关系** + **隐式视觉聚类**（前端预聚类）+ **用户手动分组覆盖**的混合策略。AI 不直接消费节点的原始几何坐标。

**当前实现状态**：仅完成"显式连接关系"消费（`collectContextText` 把 selected + connected + visible 节点的内容文本打包给 LLM）。视觉聚类预处理已在 v1 路线图（详见思考演化轨迹 1）。

**否掉的方案 A — 直接传节点几何坐标给 LLM**：
LLM 对二维空间的语义理解能力弱（区分不了 L 型布局和平行布局，区分不了"在左上"和"在右下"对设计意图的差异）。把 (x, y) 坐标和包围盒序列化进 prompt 只是徒增 token，回答质量不会有可观测提升。

**否掉的方案 B — 截屏 + Vision Model 直接看画布**：
- 每次互动多一次贵的 vision API 调用（成本高 5-10 倍）
- 画布缩放/平移状态扰动结果不可重复
- 多人协同时不同人看到不同视口，AI 上下文不一致
- 真正的语义不在像素里，在结构里——vision model 是绕远路

**底层判断**：AI 不是要"看懂画布"，是要"读懂建筑师在画布上建立的结构"——结构 = 显式连接 + 隐式聚类 + 用户主动分组，**越显式的部分越优先消费**。

**当时不确定**：纯几何聚类会不会把"住宅参考图"和"城市平面图"这种临时拼贴的内容错聚成一组。

**事后看**：基本对的——但聚类必须配合 LLM 语义化贴标，纯几何远远不够。这一层 LLM 的存在让聚类容错成本降低。

### D5 — AI 入口形态：永久 sidebar + Space 双触发

**决策**：画布右侧常驻 AI sidebar，**默认开启**，**可手动收起**；同时支持选中节点 + Space 键的快速触发（fast path）。

**否掉的方案 A — 隐藏式入口（参考 Notion AI / Miro AI）**：
AI 收进角落小图标或快捷键唤起。否掉理由：用户**第一眼想不到 AI 存在**——一旦 AI 是个"主动想起来才用"的功能，产品就降级成"画布 + AI 附加"，违反产品定位（"AI 原生 = 没有 AI 这块画布根本不成立"）。

**否掉的方案 B — 只有 Space 触发（极简 fast path）**：
Fast path 适合熟手，但新用户第一次进产品如果没人告诉他按 Space，就完全不知道有 AI。常驻 sidebar 是入门路径，Space 是熟手快捷键，二者必须共存。

**底层判断**：**AI 原生 vs AI 附加**的判断标准是——用户第一眼能否看到 AI 的存在。隐藏 = 附加，常驻 = 原生。"默认开启"声明立场，"可收起"证明诚意（尊重建筑师专注画图时的需求）。

**当时不确定**：永久 sidebar 会不会挤压画布空间影响建筑师的视觉思考。

**事后看（待长时间使用验证）**：宽屏环境下挤压可控；窄屏可能需要响应式收起策略。

### D7 — 图像模型选型：Nano Banana

**决策**：图像生成接 Nano Banana 作为默认模型；为下一阶段 multi-model 路由保留接口。

**否掉的方案 A — Midjourney / Flux 1.1 Pro**：
风格化能力强，但 prompt 服从度和结构保真度在建筑场景下不够稳定。建筑师对"原平面图的房间结构和道路被改动"是零容忍的，这一点 Midjourney 经常踩雷。

**否掉的方案 B — DALL-E 3 / GPT-4 image**：
建筑专业输出质量明显落后一档；Gemini Image 类似。

**否掉的方案 C — SD + ControlNet 自建工作流**：
最强的结构保真度（ControlNet 直接以平面图为条件输入），但工程复杂度高、推理时间长、不适合在产品 v0 阶段作为默认路径。**保留为未来 multi-model 路由的特殊任务备选**——T1（黑白平面渲染）任务可能需要它。

**底层判断**：v0 阶段需要的是"在多数任务上稳定能用"的单一模型，不是"在某一类任务上最强"的专家模型。Nano Banana 在建筑专业场景的综合表现最稳定。

**当时不确定**：是否在某些任务（特别是 T1 平面渲染）上 Nano Banana 会败给 SD+ControlNet。

**当前验证状态（诚实）**：选型基于 ~10 次定性对比 + 米理 3 位建筑专业同学的非正式反馈（差距大到无需正式打分）。**正式 benchmark 已设计完毕（附录 C），将于 [日期] 执行：8 任务 × 4 模型 × 3 评分者双盲**。

**事后看**：决策方向是对的，但定性判断必须用正式 benchmark 校准——尤其是 multi-model 路由的边界。

### D13 — AI 输出形态：直接改画布

**决策**：AI 输出 = 一个结构化操作流（`{ reply, operations[] }`），AI 可以直接增删改画布上的节点，而不是只返回文字让用户复制粘贴。

**否掉的方案 A — 纯文字回复（ChatGPT 网页版形态）**：
用户拿到 AI 的回答还要自己复制粘贴回画布。否掉理由：**违反产品根本定位**。"建筑师的 Cursor，AI 跟你一起工作"如果还要复制粘贴，那就是没消除我们立项时要消除的那个凑合方案。

**否掉的方案 B — Cursor 风格的 diff/apply 模式**：
AI 提案改动 → 用户审批 → 才生效。否掉理由：画布操作的撤销成本远低于代码修改（Ctrl+Z 一下就回来），让 AI 直接执行 + 用户事后撤销，比让用户每次审批更顺。代码改错可能误删几十行难恢复，画布改错就是几个节点的事。

**底层判断**：撤销成本决定审批粒度。代码 → 改前审批；画布 → 改后撤销。

**当时不确定**：AI 改错了用户来不及反应怎么办。

**事后看**：解决方案是把每次 AI 操作打包成一个 atomic undo 单元——一个 Ctrl+Z 就回到 AI 改之前的状态。这个细节在前端实现里必须严格遵守。

### D14 — AI 上下文范围：选中 + 连接 + 可见，不喂全画布

**决策**：AI 默认上下文 = `selectedNodes` + `connectedNodes` + `visibleNodes` 的内容文本（`collectContextText` 函数），**不把整张画布所有节点序列化喂给 LLM**。

**否掉的方案 A — 整张画布全节点喂给 LLM**：
- Token 成本爆炸（一张充实的画布几十上百节点）
- **上下文腐烂（context rot）**：LLM 注意力被无关节点稀释，回答质量下降
- 用户其实只在思考当前选中的几个节点，全喂等于不让 AI 知道"用户正关注什么"

**否掉的方案 B — 只读选中节点本身，不读连接**：
用户建立的连接关系是显式的语义信号，是用户已经亲手表达过的"这些东西相关"。不读连接 = 浪费用户已经表达的思考结构。

**底层判断**：**上下文质量 > 上下文数量**。Token 不是问题，**注意力**才是问题。

**当时不确定**：这个范围对长链条的复杂项目（节点数 50+）是否够用。

**事后看**：基本对的，但需要补"用户能手动指定上下文范围"作为 escape hatch（v1 路线图）——例如用户能框选一组节点说"AI 重点看这些"。

## 附录 C：建筑图像 Benchmark v0.1（待执行）

### 目标

用最低投入产出可信的模型选型证据，校准 D7 决策；同时为产品的 multi-model 路由路线图提供数据基础。

### 任务集（8 个 fixed cases）

| 任务 | 描述 |

| T4a / T4b | 已有渲染图 → 风格变化（日景/夜景）（×2） |

### 模型对照（4 个）

- Nano Banana（默认选型）
- Midjourney v7
- Flux 1.1 Pro
- SD + ControlNet（专业建筑工作流，特别测 T1）

### 评测维度（4 个，每维 1-5 分）

1. **结构保真度** — 场地/建筑/房间布局有没有错位
2. **材质光照真实感** — 渲染图的物理可信度
3. **prompt 服从度** — 要求的元素都出现了吗
4. **风格一致性** — 同任务多次输出风格是否稳定

外加一个布尔指标：**绝对崩坏率** — 多少次输出完全不可用。

### 评测方法

- **双盲**：同一组 input 跑所有模型，输出图随机命名 m1.png / m2.png ...
- 评分者：本人 + 米理 2 位建筑专业同学（共 N=3）
- 每图 4 维 × 1-5 分独立打分，再合并
- 计算评分者间一致性（IRR）—— 建筑师对建筑图的判断分歧本身就是一个值得呈现的事实

### 投入估算

- 实做：3-4 小时（生成所有图 + 准备评分表）
- 评分者：3 人 × 30 分钟
- 分析与雷达图：1-2 小时
- **总计：周末半天**
